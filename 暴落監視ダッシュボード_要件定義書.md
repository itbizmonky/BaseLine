# 暴落対応ルール監視ダッシュボード 要件定義書

**作成日：** 2026年7月
**開発環境：** Google AntiGravity
**対象：** 新NISA攻撃フェーズ 階層型トリガー監視の自動化

---

## 1. 背景・目的

### 1.1 背景
新NISA攻撃フェーズにおいて、以下4銘柄の暴落対応ルール（階層型トリガー）を毎日目視監視する運用としている。

| 銘柄 | Tier1 | Tier2 | Tier3 |
|---|---|---|---|
| iFreeNEXT FANG+インデックス | -15% | -25% | -35% |
| ニッセイSOX指数インデックスファンド | -10% | -18% | -28% |
| eMAXIS Slim米国株式(S&P500) | -7% | -12% | -18% |
| eMAXIS Slim全世界株式(オルカン) | -6% | -10% | -15% |

現状は人手による毎日の確認が前提だが、**見逃しリスクの排除**と**運用負荷ゼロ化**を実現するため、自動監視ツールを自作する。

### 1.2 目的
- 毎朝ワンクリック（またはアクセスのみ）で4銘柄の現状を一目で把握できること
- トリガー条件に合致した場合、即座に通知を受け取れること
- 監視自体は人手を介さず、クラウド上で自動的に毎日実行されること

## 2. 対象銘柄・トリガー条件（再掲）

| No | 銘柄 | 判定基準 | Tier1 | Tier2 | Tier3 |
|---|---|---|---|---|---|
| 1 | iFreeNEXT FANG+インデックス | 設定来高値比 | -15% | -25% | -35% |
| 2 | ニッセイSOX指数インデックスファンド | 設定来高値比 | -10% | -18% | -28% |
| 3 | eMAXIS Slim米国株式(S&P500) | 設定来高値比 | -7% | -12% | -18% |
| 4 | eMAXIS Slim全世界株式(オール・カントリー) | 設定来高値比 | -6% | -10% | -15% |

**設定来高値の定義：** `config/settings.json`の`peak_start_date`（2026-07-07。購入判定基準日`baseline.date`と同一日に設定）以降に記録した各銘柄の基準価額の最高値。日次で自動更新（当日値が過去の記録より高ければ更新）。
※ピーク追跡開始日（`peak_start_date`）と、下記の資金投入期間（②期間開始日）は別の設定値であり、意図的に分離されている。ピーク追跡は2026年7月7日から開始するが、実際の資金投入は②期間（2026年8月〜）から。そのため②期間開始前でもTier到達・LINE通知は発生し得る。

**監視対象期間：**
- ②資金トリガー：2026/08〜2027/02末（原則）
- ③資金トリガー：2027/03〜2027/08末（原則）
- 両期間とも、未消化の場合は2027/12末まで延長可（例外）

## 3. システム構成

### 3.1 全体アーキテクチャ

```
[GitHub Actions（定時実行トリガー）]
        ↓ JST平日朝7:00頃（settings.jsonで変更可、cronはUTC・JSTの曜日ズレを補正した値を自動生成。実際の着信はGitHub Actions側の遅延により前後する）
[Pythonスクリプト：データ取得・判定]
        ↓
   ①日本経済新聞 電子版（日経電子版）投資信託ページをスクレイピング
   ②基準価額を取得・記録DBに保存
   ③設定来高値（2026年8月以降のみ記録）と比較し下落率を計算
   ④Tier判定・購入判定（BUY/WAIT/HOLD/HIGH）
        ↓
   [判定結果をHTMLダッシュボードとして生成・GitHub Pagesに公開]
        ↓
   [Tier到達時のみ：LINE Notify APIで即時通知]
        ↓
[ユーザー：スマホ/PCブラウザでダッシュボードURLにアクセス → 一目で確認]
```

### 3.2 なぜこの構成か
- **GitHub Actions採用理由：** 無料枠（public repoなら無制限、private repoでも月2,000分無料）でPC電源に依存せず定時実行できる
- **GitHub Pages採用理由：** 生成したHTMLを無料で公開でき、スマホからブックマーク1つでアクセス可能
- **LINE Notify採用理由：** メール以外の通知手段として、スマホへのプッシュ通知が最も即時性が高い

## 4. 機能要件

### 4.1 データ取得機能

| ID | 要件 | 詳細 |
|---|---|---|
| F-01 | 基準価額の自動取得 | 4銘柄の日次基準価額を日経電子版の投資信託ページからスクレイピングで取得 |
| F-02 | 取得失敗時のリトライ | スクレイピング失敗時は3回まで自動リトライ（間隔30秒）、それでも失敗した場合はエラー通知を送信 |
| F-03 | 取得データの永続化 | 取得した基準価額・日付をCSVまたは軽量DB（SQLite等）としてリポジトリ内に保存し、履歴を蓄積 |
| F-04 | 設定来高値の自動更新 | 当日の基準価額が保存済みの最高値を上回った場合、来高値を自動更新。初回記録時（`peak_start_date`を過去日付に設定した場合を含む）は、`history.csv`の`peak_start_date`以降の記録も遡って確認し、本日の値より高い記録があればそちらを初期値として採用（取りこぼし防止） |

### 4.2 判定機能

| ID | 要件 | 詳細 |
|---|---|---|
| F-05 | 下落率の自動計算 | 下落率 ＝ (設定来高値 − 当日基準価額) ÷ 設定来高値。負記号「-」を用いてパーセント表示（現在値が設定来高値以上の場合は0%とし、負の下落率は発生しない） |
| F-06 | Tier判定 | 各銘柄の下落率を、Tier1/2/3の閾値と比較し、到達しているTierを判定（未到達時は「未到達」と表示） |
| F-06b | 購入判定（BUY/WAIT/HOLD/HIGH） | Tier到達に加え、現在価格が基準日価格（`settings.json`の`baseline`、許容上昇率デフォルト5%）以内かどうかで BUY／WAIT を判定。高値更新中は HIGH、Tier未到達は HOLD。基準日価格は固定値であり自動更新は行わない（現状の仕様として維持） |
| F-07 | 既発動Tierの記録 | 一度発動したTierは「発動済み」として記録し、同一Tierでの重複通知を防止 |
| F-08 | 期間内判定 | ②③のどちらの資金期間内かを日付で自動判別。開始前は「②期間開始前」、終了後は「監視期間終了」と識別（ピーク追跡自体は`peak_start_date`から別途開始しているため、②期間開始前でもTier到達・通知は発生し得る点に注意） |

### 4.3 通知機能

| ID | 要件 | 詳細 |
|---|---|---|
| F-09 | Tier到達時の即時通知 | いずれかの銘柄が新たにTierに到達した場合、当日の実行タイミングでLINE Messaging APIへ通知 |
| F-09b | 日次サマリー通知 | 平日毎朝の実行時に、Tier到達の有無にかかわらず4銘柄の状況（下落率・基準日比・判定）をまとめてLINE Messaging APIへ通知 |
| F-10 | 通知内容 | 銘柄名／到達Tier／下落率／投入予定額／実行期間（②or③）を含む |
| F-11 | データ取得エラー時の通知 | スクレイピング失敗が続いた場合、別途エラー通知を送信し「本日は確認できていない」ことを明示 |

### 4.4 ダッシュボード表示機能

| ID | 要件 | 詳細 |
|---|---|---|
| F-12 | 一覧表示 | 4銘柄を1画面に並べ、各銘柄の「現在値」「設定来高値」「下落率」「Tier到達状況」を一目で確認できる |
| F-13 | 視覚的な状態表示 | 下落率に応じて色分け（緑：Tier未到達／黄：Tier1到達／橙：Tier2到達／赤：Tier3到達） |
| F-14 | 推移グラフ | 各銘柄の過去の基準価額推移と、設定来高値・各Tier閾値ラインを重ねた折れ線グラフを表示 |
| F-15 | 残存資金の表示 | ②③それぞれの原資のうち、各Tierで投入済み／未投入の金額を表示 |
| F-16 | 期間表示 | 現在が②期間か③期間か、期限まで残り何日か、または②期間開始前・終了後であるかを表示 |
| F-17 | 簡易予想表示 | 直近5営業日・20営業日のトレンド（上昇/下落/横ばい）を機械的に算出し表示（複雑な予測モデルは含まない） |

## 5. 非機能要件

| ID | 要件 | 詳細 |
|---|---|---|
| NF-01 | 運用不要 | 一度セットアップした後、人手による日次操作は一切不要（GitHub Actionsが自動実行） |
| NF-02 | 実行タイミング | 平日のみ、日本時間 朝7:00頃に自動実行（`settings.json`の`schedule`で変更可）。GitHub Actionsのschedule実行は無料枠では正確な時刻を保証されず、実際の着信は30〜60分程度遅れることがある（既知の制約として許容）。※2026-07調査により、日経電子版の基準価額は当日夜に確定・反映されるため、朝実行時点では前営業日分の確定値を取得する仕様であることが判明（不具合ではなく情報源の公表タイミングに起因する仕様）。詳細は改修履歴を参照 |
| NF-03 | 可用性 | GitHub Actions・GitHub Pagesとも無料枠内での運用を前提とし、追加費用が発生しないこと |
| NF-04 | 見やすさ | スマートフォンの小さい画面でも1画面で全4銘柄の状態が把握できるレスポンシブ対応 |
| NF-05 | 取りこぼし防止 | 平日は必ず実行される設計とし、土日はスキップ（投信基準価額が更新されないため）。日付・曜日判定はすべてJST基準で行い、GitHub Actions実行環境（UTC）とのタイムゾーン差による日付ずれ・週末重複記録を防止する（祝日は未対応） |
| NF-06 | 保守性 | Tier閾値・監視対象銘柄・原資金額は設定ファイル（JSON等）で管理し、コード変更なしで調整可能にする |

## 7. 開発スコープ（Google AntiGravityへの依頼内容）

### 7.1 含めるもの
- Yahoo!ファイナンス投信ページのスクレイピングスクリプト（Python）
- 判定ロジック（下落率計算・Tier判定・期間判定）
- LINE Notify連携（Messaging APIまたはLINE Notify後継サービス）
- GitHub Actions用のワークフロー定義（YAML）
- GitHub Pages公開用のHTMLダッシュボード生成スクリプト
- 設定ファイル（銘柄コード・Tier閾値・原資金額をJSON管理）

### 7.2 含めないもの（スコープ外）
- 実際の買付注文の自動発注（SBI証券での注文は引き続き手動）
- 高度な株価予測AI（トレンド表示は単純な移動平均比較にとどめる）
- 複数ユーザー対応・ログイン機能（個人利用前提）

## 8. 残課題・確認事項

| No | 内容 | 状態 |
|---|---|---|
| 1 | データ取得元のスクレイピングが規約上問題ないか | ✅解決済み。実装では日経電子版（当初想定のYahoo!ファイナンスから変更）を個人の非営利利用・低頻度アクセス（1日1回）で利用。日経電子版側の利用規約の詳細確認は未実施（残課題） |
| 2 | 各銘柄の正確な銘柄コード | ✅解決済み。`config/settings.json` の `fund_code` に日経電子版の銘柄コードとして記録済み |
| 3 | LINE Notifyのサービス終了に伴う代替 | ✅解決済み。LINE Messaging API（Push通知）で実装済み |
| 4 | 祝日（土日以外の休場日）の扱い | 未対応。現状は土日のみ自動スキップし、祝日は通常通り実行される（休場のためスクレイピング結果が前営業日と同値になる） |
| 5 | 購入判定（BUY/WAIT）の基準日価格が固定である点 | 認識済み・現状維持の判断（2026-07レビュー時に確認）。基準日価格は自動更新されないため、長期的な相場上昇局面ではTier到達時にもWAIT判定が続く可能性がある |

## 9. 開発ステップ（想定）

1. 日経電子版から対象4銘柄の基準価額を取得するスクレイピング処理を実装・動作確認
2. 下落率計算・Tier判定ロジックを実装
3. LINE Messaging API連携で通知テストを実施
4. HTMLダッシュボードのテンプレートを作成
5. GitHub Actionsワークフローを設定し、手動トリガーでの動作確認
6. 定時実行（cron）を設定し、数日間の自動実行をテスト
7. 本番運用開始

## 10. 改修履歴

| 時期 | 内容 |
|---|---|
| 2026-07 | Claude Codeへ開発移管。コードレビューにより以下の不具合を発見・修正：①`date.today()`がUTC基準で日付・曜日判定を行っており、JST実行時刻との間で日付が1日ずれる不具合、②`schedule.daily`が`true`のまま運用されており土日も実行され、休場日のNAVが別日付で重複記録される不具合、③監視開始前（`peak_start_date`未到達）にもかかわらず設定来高値が仮登録され、その後NAVがそれを上回ると下落率が負値になりLINE通知が「--1.1%」のような二重マイナス表記になる不具合。いずれも修正済み（[scripts/monitor.py](scripts/monitor.py), [scripts/judge.py](scripts/judge.py)）。あわせて`.gitignore`整備、判定表示ロジックの共通化、ダッシュボードの表記修正（データ出典・HIGH配色）を実施。GitHub Actions本番実行によりLINE通知・ダッシュボード反映まで動作確認済み |
| 2026-07 | `peak_start_date`を`2026-08-01`（②期間開始日）から`2026-07-07`（購入判定基準日`baseline.date`と同一日）に変更。ユーザー判断により、②期間開始前から高値追跡を行う運用に変更。`peak_start_date`を過去日付に設定してもhistory.csvの記録済み履歴を遡って確認し、取りこぼしなく初期ピークを設定する処理を追加（`judge.seed_initial_peak()`） |

| 2026-07 | 実行時刻をJST 07:00から07:15に変更（ユーザーの私的な確認しやすい時刻に合わせた設定変更） |
| 2026-07 | 朝の本番実行結果をユーザーが確認したところ、①GitHub Actionsのschedule実行が予定時刻から55分遅延（08:10着信）、②オルカンが基準日価格を下回ったままなのに「本日高値更新（HIGH）」と誤判定、という2点が判明。①はGitHub側の既知の仕様のため実行時刻を07:15→07:00に戻し遅延を吸収する運用とした。②は`update_peak()`が`history.csv`の欠落期間（2026-07-07〜09、監視ツール導入前で未記録）を考慮しておらず、`settings.json`の`baseline.prices`（基準日実績値）を見逃していたことが原因と判明。基準日価格を来高値の下限として毎回確認する自己修復ロジックを追加し修正。あわせてLINE日次サマリーの「最高値比」「基準日比」を改行表示に変更、期間ラベル「監視開始前」を「②期間開始前」に変更しF-08/F-16の記述も追従 |
