# BaseLine ドキュメント同期

BaseLineは個人用の暴落監視ダッシュボードで、コード・設定の変更が積み重なるとドキュメントとの乖離が生じやすい（実際に過去、データ取得元がYahoo!ファイナンスから日経電子版に変わった後もドキュメントが古いままだった、実行時刻を変更した後にREADMEの表記が古いままだった、という乖離が複数回発生している）。このスキルは、コード・設定変更のたびに関連ドキュメントを機械的に追従させるための手順を示す。

## いつ確認するか

以下のいずれかを行った直後は、必ず3ドキュメントへの影響を検討する。

- `scripts/*.py` のロジック変更（判定ロジック、通知内容、データ取得方法など）
- `config/settings.json` の変更（スケジュール、Tier閾値、期間、ピーク追跡開始日など）
- `.github/workflows/monitor.yml` の変更
- ユーザーとの会話で判明した設計判断・不具合調査の結果（コード変更を伴わない場合でも、将来のセッションで再度同じ調査をしなくて済むよう記録する価値がある）

変更が軽微（typoやコメント修正など、ユーザーに影響しないもの）であれば、ドキュメント更新は不要と判断してよい。

## どのドキュメントに何を反映するか

3ファイルは役割が異なるため、同じ変更でも書く内容・書く場所が違う。

| 変更の種類 | README.md | 要件定義書 | SESSION_CONTEXT.md |
|---|---|---|---|
| スケジュール・実行時刻の変更 | 概要表・通常運用セクションの時刻表記 | NF-02（実行タイミング）、処理フロー図の時刻表記 | 現在の状態＋セッション履歴に追記 |
| 判定ロジック・通知内容の変更 | 該当する説明文（あれば） | 4.2/4.3の該当F-ID（例: F-06b, F-09b）、なければ新規F-IDを採番して追加 | 現在の状態＋セッション履歴に追記 |
| データ取得元・外部サービスの変更 | 注意事項セクション | 3.1アーキテクチャ図、F-01、8章の残課題 | 現在の状態＋セッション履歴に追記 |
| 設定ファイルの新規項目追加 | カスタマイズセクション | 該当する非機能要件（NF-06関連） | セッション履歴に追記 |
| 不具合調査・原因究明（コード変更なしの場合を含む） | 通常は不要 | 8章の残課題、または10章改修履歴に調査結果を1行で記録 | セッション履歴に必ず記録（同じ調査を将来繰り返さないため） |

判断に迷ったら、「このドキュメントだけを読んだ人が、今の実装を正しく理解できるか」を基準に考える。

## 各ファイルの編集ルール

**README.md・要件定義書**：該当箇所を直接書き換える（上書き更新）。要件定義書の末尾「10. 改修履歴」テーブルだけは例外で、既存行を消さずに新しい行を追加する。1行の内容は「時期｜何を・なぜ変更したか」を簡潔に。

**SESSION_CONTEXT.md**：2つのセクションで扱いが異なる。
- 「現在の状態」：常に最新の状態を反映するよう、古い記述を書き換える（プロジェクトの"今"のスナップショット）。
- 「セッション履歴」：**追記専用**。過去のエントリは絶対に削除・書き換えしない。今回の変更について新しい日付見出しの下に、何を・なぜ・どう検証したかを記録する。次のセッションの自分（または別のセッション）が、このファイルの「セッション履歴」だけを読めば経緯を追えるように書く。

## やらないこと

- `public/index.html` は`generate_dashboard.py`が生成する成果物であり、ドキュメントではない。手動編集しないこと（次回実行で上書きされるため無意味）。
- `CLAUDE.md` はコーディング規約・開発ガイド用のファイルで、このスキルの対象外（内容が大きく変わった場合のみ別途検討する）。
- 3ファイルすべてを毎回機械的に全部更新する必要はない。上記の対応表を参考に、実際に影響する箇所だけを更新する。
